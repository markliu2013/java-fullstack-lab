# 并发模型经典题：交替打印
使用两个线程：
- 线程 T1：输出数字 1~26
- 线程 T2：输出字母 A~Z

要求最终输出结果为： 1A2B3C4D...26Z

抽象问题 → 找约束 → 定义模型 → 再选手段

第一步：先把表象撕掉 抽象问题
❌ 打印什么不重要
❌ 是数字还是字母不重要
❌ 是不是两个线程也不本质

真正抽象后只有一句话
多个执行体按严格顺序轮流获得执行权
这一步抽象非常关键

第二步：把问题转成“执行权模型”
想象一根接力棒（这是面试官脑子里的模型）：
A 拿到执行权 → 执行 → 把执行权交给 B
B 拿到执行权 → 执行 → 把执行权交给 A
注意这里的关键词：
执行权（Execution Token）

本质不是“交替打印”
而是：
执行权在不同线程之间按既定顺序流转

第三步：列出系统必须满足的约束
约束 1：互斥执行
任意时刻只能有一个线程打印。

约束 2：严格顺序（强时序约束）
不是“谁先抢到谁执行”，而是顺序被写死：

约束 3：执行权必须可靠传递（不能丢）
这是这题最隐蔽、但最核心的本质。
每一轮必须保证：
A 执行完 → B 一定能被唤醒
B 执行完 → A 一定能被唤醒
不能出现：
信号丢失
双方都睡死
永久等待

约束 4：系统效率要求（隐藏但重要）
一个工业级方案必须：
不能忙等
不能空转 CPU
不能 sleep 猜时间
要可扩展
这其实是面试官在区分：
能写题的人
能写系统的人

第四步：把问题升维成经典并发模型
线程之间如何“协作”
生产者-消费者
接力棒（baton passing）
执行权必须被精确地 handoff 给下一个线程

这道题的本质是多个线程之间基于确定性顺序的执行权传递问题，需要在满足互斥和条件约束的前提下，实现可靠的线程间 handoff，并保证系统不会退化为忙等。

扩展问题：
多线程变成单线程了，为什么不直接用单线程呢？
